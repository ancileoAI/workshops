# Covariance, Contravariance, and Invariance

## Mathematical Foundation

At its core, variance describes how **type transformations** affect the **subtype relation**. Given:
- `A` and `B` are types  
- `f` is a type transformation (like `List<T>`, `T[]`, or `Callable[[T], R]`)
- `≤` (or `<:`) represents the subtype relation (`A ≤ B` means "A is a subtype of B")

We formally define:

```
f is COVARIANT     if A <: B implies f(A) <: f(B)     [preserves direction]
f is CONTRAVARIANT if A <: B implies f(B) <: f(A)     [reverses direction]  
f is INVARIANT     if neither of the above holds      [no relationship]
```

## 1. Covariance: "Preserves the Flow"

### Definition
Generic type `GenType[T]` is **covariant** in type variable `T` if:
```
GenType[SubType] <: GenType[SuperType], for SubType <: SuperType
```

### Examples

#### Python Tuples (Immutable)
```python
class Animal: pass
class Dog(Animal): 
    def bark(self): pass

# This works - covariance in action!
dogs: Tuple[Dog, ...] = (Dog(), Dog())
animals: Tuple[Animal, ...] = dogs  # ✅ Safe assignment

# This doesn't work
animals: Tuple[Animal, ...] = (Dog(), Animal())
dogs: Tuple[Dog, ...] = animals  # ❌ Error: animals might contain non-Dogs
```

**Why it's safe**: Every `Dog` is an `Animal`, and tuples are immutable, so no new elements can be added that would violate the type constraint.

#### Java Arrays (Covariant but Unsafe!)
```java
// Given String <: Object
String[] strings = {"hello", "world"};
Object[] objects = strings;  // ✅ Compiles (arrays are covariant)

// But this can cause runtime errors:
objects[0] = 42;  // ❌ Runtime error: ArrayStoreException
```

#### Function Return Types
```python
from typing import Callable

def get_dog() -> Dog: return Dog()
def get_animal() -> Animal: return Animal()

# Covariant in return type
animal_getter: Callable[[], Animal] = get_dog  # ✅ Safe
# dog_getter: Callable[[], Dog] = get_animal   # ❌ Unsafe
```


**Rule**: Producers (read-only, factories, getters) are covariant in what they produce.

## 2. Contravariance: "Reverses the Flow"

### Definition
Generic type `GenType[T]` is **contravariant** in type variable `T` if:
```
GenType[SuperType] <: GenType[SubType], for SubType <: SuperType
```

### Examples

#### Function Parameters
```python
def feed_animal(animal: Animal) -> None: 
    print(f"Feeding {animal}")

def feed_dog(dog: Dog) -> None: 
    print(f"Feeding dog: {dog}")
    dog.bark()

# Contravariant in parameter types
dog_feeder: Callable[[Dog], None] = feed_animal  # ✅ Safe
# animal_feeder: Callable[[Animal], None] = feed_dog  # ❌ Unsafe
```

**Why it's safe**: If you need a function that can handle Dogs, a function that can handle any Animal will work fine. But a Dog-specific function can't safely handle arbitrary Animals.

#### Real-world Example: Event Handlers
```python
class UIEvent: pass
class ClickEvent(UIEvent): 
    def get_coordinates(self): pass

def handle_any_ui_event(event: UIEvent) -> None:
    print("Handling UI event")

def handle_click_event(event: ClickEvent) -> None:
    print(f"Click at {event.get_coordinates()}")

# This assignment is safe due to contravariance
click_handler: Callable[[ClickEvent], None] = handle_any_ui_event  # ✅
```

**Rule**: Consumers (write-only, processors, setters) are contravariant in what they consume.

## 3. Invariance: "No Relationship"

### Definition
Generic type `GenType[T]` is **invariant** in type variable `T` if it's neither covariant nor contravariant:
```
Neither GenType[SubType] <: GenType[SuperType] 
nor GenType[SuperType] <: GenType[SubType]
```

### Examples

#### Python Lists (Mutable)
In contrast with Java python Lists Are NOT Covariant. In Python, the type system treats lists as invariant, not covariant:

```python
from typing import List

class Animal: pass
class Dog(Animal): pass

dogs: List[Dog] = [Dog(), Dog()]
animals: List[Animal] = dogs  # ❌ mypy error: incompatible types
```

However, Python's runtime behavior is different because Python is dynamically typed:

```python
# This runs fine at runtime (no type checker involved)
dogs = [Dog(), Dog()]
animals = dogs  # Same object reference
animals.append(Cat())  # No runtime error - Python doesn't care about types
print(dogs)  # Now contains a Cat!
```

#### Java Generics
```java
// These don't compile (generics are invariant):
ArrayList<String> strings = new ArrayList<Object>();  // ❌
ArrayList<Object> objects = new ArrayList<String>();  // ❌

// Only this works:
ArrayList<String> strings = new ArrayList<String>();  // ✅
```

**Rule**: Mutable containers (read-write) are invariant because they're both producers AND consumers.

## 4. Practical Applications

### Assignment Rules
The statement `x = y;` compiles only if `typeof(y) ≤ typeof(x)`.

```java
// Method signature:
Number[] method(ArrayList<Number> list) { ... }

// These don't compile due to invariance:
Integer[] result = method(new ArrayList<Integer>());  // ❌
Object[] result = method(new ArrayList<Object>());    // ❌

// These work:
Number[] result = method(new ArrayList<Number>());    // ✅
Object[] result = method(new ArrayList<Number>());    // ✅ (covariant return)
```

### Method Overriding Rules
For safe method overriding:
- **Parameter types** must be **contravariant** (or invariant)
- **Return types** must be **covariant** (or invariant)

```java
class Super {
    Number method(Number n) { ... }
}

class Sub extends Super {
    @Override 
    Integer method(Number n) { ... }  // ✅ Covariant return type (since Java 1.5)
    // Integer method(Integer n) { ... }  // ❌ Would be contravariant parameter
}
```

**Runtime explanation**: When calling `super.method()`, it actually calls `sub.method()`. For this to be safe:
```java
// This substitution must work:
Number result = super.method(someNumber);
// Becomes:
Number result = sub.method(someNumber);  // sub's method must accept Number (contravariant)
                                        // sub's method can return Integer (covariant)
```

## 5. Memory Aids

### PECS Principle (Producer Extends, Consumer Super)
- **Producers** should be **covariant** (`? extends T`) - they only give you things
- **Consumers** should be **contravariant** (`? super T`) - they only take things  
- **Both** should be **invariant** - they both give and take

### Real-world Analogies
- **Covariant**: A dog factory can substitute for an animal factory (every dog produced is an animal)
- **Contravariant**: An animal shelter can substitute for a dog shelter (it can handle dogs just fine)
- **Invariant**: A dog kennel can't substitute for an animal kennel (you might put cats in, or take out non-dogs)

## 6. Language Comparisons

| Language | Arrays | Generics | Function Returns | Function Parameters |
|----------|--------|----------|------------------|-------------------|
| Java     | Covariant (unsafe) | Invariant | Covariant | Contravariant |
| C#       | Covariant (unsafe) | Invariant* | Covariant | Contravariant |
| Python   | N/A | Invariant* | Covariant | Contravariant |
| TypeScript | Covariant | Covariant (unsafe) | Covariant | Contravariant |

*With explicit variance annotations (`out`/`in` in C#, `covariant`/`contravariant` in Python)

## Summary

Variance is about **substitutability and safety**:

1. **Covariant**: Subtype relationship preserved - safe for producers/read-only structures
2. **Contravariant**: Subtype relationship reversed - safe for consumers/write-only structures  
3. **Invariant**: No subtype relationship - safe for mutable/read-write structures

Understanding variance helps you:
- Write type-safe code
- Understand compiler errors
- Design better APIs
- Reason about generic type relationships

The key insight: variance depends on how the generic type is **used**, not just how it's **defined**.
